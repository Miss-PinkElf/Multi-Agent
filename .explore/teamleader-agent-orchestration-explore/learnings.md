# 经验沉淀

## 有效做法
- 先把问题拆成控制平面、执行平面、通信层、状态层、并行控制层，再讨论具体工具，能显著减少选型噪声。
- 对这类长任务系统，真正关键的是“恢复、冲突控制、人工接管”，不是单轮模型表现。
- 把 UI、协议、状态真相源分层看，比“找一个全包框架”更容易做出稳定系统。

## 无效做法
- 直接把多 agent 框架当最终系统设计，容易忽略长期运行、恢复和冲突处理。
- 用 star 数单独判断项目价值不可靠；更重要的是项目是否已经实现关键骨架能力。

## 降低成本或延迟的技巧
- 执行层先用 shell/tmux/worktree，本地闭环跑通后再接 ACP 或更复杂的 agent 协议。
- 让 leader 用代码控制流程，worker 再交给模型自由发挥，通常比全链路靠模型决策更省且更稳。

## 可复用策略
- 用 `git worktree + file/task locks + merge gate` 解决并行代码冲突。
- 用 `SQLite + JSONL` 同时满足事务状态和可读事件日志。
- 用 append-only event log 管理 agent 生命周期，更容易做监控、回放和 handoff。

## 要避免的坑
- 一开始就做成“大而全平台”，会把任务拖进 UI、协议、执行器和存储同时开工的复杂度。
- 不区分“任务冲突”和“代码冲突”，会导致并行看似可行但集成阶段爆炸。
- 没有统一的 handoff 结构，长上下文恢复会高度依赖聊天记录，难以稳定续接。

## 完整记录

### [#1] 2026-04-02
- 决策：把目标重新定义为“外层 orchestrator / Team Leader”，而不是“再加一个会写代码的 agent”。
- 原因：用户关心的是调度、监控、恢复、仲裁、冲突控制和交接，而不是单个 agent 的编码能力。

### [#2] 2026-04-02
- 决策：不把 Claude Code agent teams 作为最终底座，只把它当参考方案。
- 原因：其更接近实验性能力，且成本、可视化、恢复与长期运行体验不完全满足目标。

### [#3] 2026-04-02
- 决策：把 Zed 定位为观察和人工接管界面，而不是控制平面。
- 原因：Zed 适合作为 IDE 和外部 agent 接入层，但不适合承担完整的调度、状态真相源和冲突控制职责。

### [#4] 2026-04-02
- 决策：优先把 ACP / acpx 看成通信层，而不是完整解决方案。
- 原因：协议和 CLI 能帮助结构化会话控制，但不直接解决调度、锁、handoff、优先级和状态存储。

### [#5] 2026-04-02
- 决策：优先推荐“借鉴架构，自写新壳”的路线，而不是重度 fork 单个开源项目。
- 原因：这样能复用成熟思路，同时避免被某个项目的内部抽象和生命周期绑定。

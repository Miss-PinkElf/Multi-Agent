## 关键决策与原因

| 决策 | 备选方案 | 原因 |
| --- | --- | --- |
| 把目标定义为 Team Leader / Orchestrator | 再找一个更强的 coding agent | 用户关心的是调度、监控、恢复、仲裁，而不是单体编码能力 |
| 不把 Claude Code agent teams 当最终底座 | 直接依赖 agent teams | 其更像实验性参考实现，成本与长期运行体验不完全匹配目标 |
| 把 Zed 定位为观察/接管界面 | 让 Zed 承担控制平面 | Zed 更适合 IDE 与外部 agent 接入，不适合作为唯一状态真相源 |
| 把 ACP/acpx 视为通信层 | 直接把 ACP 当完整方案 | 协议能解决会话通信，但不直接解决锁、恢复、调度和冲突控制 |
| 优先推荐“借鉴架构，自写新壳” | 重度 fork 单个项目 | 更容易保留系统主导权，避免被单个项目内部抽象绑住 |
